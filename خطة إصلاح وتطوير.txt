خطة إصلاح وتطوير (نسخة عملية قابلة للتنفيذ)
ملخص
إصلاح الأعطال التي تمنع التشغيل، ثم إصلاح تضارب قواعد Firestore مع الواجهة، ثم معالجة الأمان والأداء، وأخيرًا تحضير خارطة تطوير تدريجية باتجاه الرؤية الكبرى في dayf.txt.

خطوات التنفيذ

إصلاح أعطال البناء والتشغيل
توحيد مكتبة الحركة في جميع الملفات.
تحديث صفحة المجتمع لاستخدام user وuserProfile بدل currentUser.
مواءمة قواعد Firestore مع سلوك التطبيق
السماح بترقية المستخدم لدور provider.
السماح للمزود بتحديث حالة الحجز.
منع تعديل likesCount وrepliesCount يدويًا دون تحقق.
إصلاحات وظيفية أساسية
تواريخ الحجز تصبح مدخلة من المستخدم بدل تواريخ ثابتة.
منع المستخدم غير المصرّح له من إضافة “إعلان رسمي”.
توحيد اتجاه اللغة مع documentElement.dir.
تحسينات الأمان
نقل Gemini API إلى خادم وسيط أو Cloud Function.
تقييد استدعاءات AI وإضافة كاش أو حدود استخدام.
تحسينات الأداء وتجربة المستخدم
تقليل عدد اشتراكات Firestore في بطاقات الخدمات.
إضافة Lazy Loading للصور الثقيلة.
تفعيل PWA فعليًا أو إزالة ملفاتها بالكامل لتفادي الالتباس.
خارطة الطريق باتجاه dayf.txt
تحديد أي أجزاء من مخطط الجداول ستُنفّذ أولًا (الحجوزات، المجتمع، السوق).
تصميم نموذج بيانات Firestore أو الانتقال لقاعدة SQL إذا كانت متطلبات العلاقات معقدة جدًا.
الاختبارات المقترحة

تسجيل الدخول وتبديل اللغة.
إنشاء خدمة كمزود.
إجراء حجز وتأكيده من طرف المزود.
إنشاء موضوع مجتمع وإعجابه.
تجربة البحث الذكي والتأكد من سلامة الاستجابة.
اختبار اتجاه الصفحة بالعربية والإنجليزية.
افتراضات

الهدف الحالي هو إصلاح الـ MVP أولًا، وليس تنفيذ كامل مخطط dayf.txt دفعة واحدة.
ستبقى Firebase هي الباكند الحالي في المرحلة الأولى.
